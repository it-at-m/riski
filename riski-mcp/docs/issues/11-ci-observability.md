# CI & observability

**Labels:** `mcp`, `chore`, `ci`
**Depends on:** #06, #10

## Goal

Add CI for `riski-mcp` consistent with the repo, an optional deploy-to-HF step, and
lightweight observability so robot-driven calls are debuggable in production.

## Tasks

- [ ] CI workflow (GitHub Actions) for `riski-mcp/`: `uv sync`, `ruff check`,
      `ruff format --check`, `ty`/type check, `pytest` with coverage. Trigger on
      changes under `riski-mcp/**`.
- [x] Keep `requirements.txt` (HF) and `pyproject.toml` deps in sync. **Done in #01**:
      `requirements.txt` is auto-generated from `uv.lock` via `uv export` and
      regenerated by a pre-commit hook. CI should additionally run
      `uv export ... -o requirements.txt` and fail if it produces a diff (catches
      commits made without pre-commit).
- [ ] Optional: GitHub Action to push `riski-mcp/` to the HF Space on release/tag
      (using an `HF_TOKEN` secret), if the repo-as-source-of-truth approach (#08) is
      chosen.
- [ ] Observability:
  - Structured logging (request id, tool name, latency, backend status, error_type),
    **no secrets/PII**.
  - Log AG-UI stream outcomes (event counts, terminal state) at DEBUG.
  - Decide whether to add Langfuse parity (the backend already traces); likely
    unnecessary in the thin wrapper — document the decision.
- [ ] Basic abuse protection given the Space is public: per-IP/simple rate limit or
      lean on backend-side limits; document the chosen approach.

## Acceptance criteria

- [ ] CI runs on PRs touching `riski-mcp/` and gates merges.
- [ ] Lint, type check, and tests are green.
- [ ] Logs let you trace a single robot question through to the backend response.
- [ ] Dependency drift between `requirements.txt` and `pyproject.toml` is detected.

## Notes

- A public MCP Space is an open door to the backend — coordinate rate limiting /
  auth with the backend owners (cross-ref #08).
